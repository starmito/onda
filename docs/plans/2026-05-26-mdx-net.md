
**Requisitos funcionales del script:**
1. **Carga de modelo ONNX:** `ort.InferenceSession(model_path, providers=['CUDAExecutionProvider', 'CPUExecutionProvider'])`
2. **Pre-procesado:** STFT vía `torch.stft` con ventana Hann de 6144, hop 1024
3. **Inferencia por chunks:** dividir espectrograma en chunks de `dim_t=256` frames con overlap configurable
4. **Post-procesado:** iSTFT para reconstruir audio
5. **Modo CLI:** argparse con `--model`, `--input`, `--output`, `--overlap`, `--list-models`
6. **Modo librería:** función `separate()` importable desde otros scripts
7. **Manejo de errores:** validar que el modelo .onnx existe, que el audio es válido, logging claro

**Archivos de referencia:**
- `inference/inference_universal.py` — patrón de arquitectura (CLI + separate())
- `inference/lib_v5/mdxnet.py` — arquitectura ConvTDFNet (para entender dims esperados)
- `inference/lib_v5/spec_utils.py` — utilidades de STFT ya existentes

### PLAN DE CONTINGENCIA

**Si Task 3 falla (script no funciona):**
- Verificar que el modelo .onnx se cargó correctamente: `onnxruntime.InferenceSession` sin errores
- Verificar dimensiones de entrada: MDX-Net espera `(batch, 4, 2048, 256)` — [4 canales, 2048 freq bins, 256 time frames]
- Si hay mismatch de dimensiones → ajustar N_FFT/DIM_F/DIM_T según metadata del modelo

**Si Task 5 falla (inferencia en contenedor):**
- Verificar que onnxruntime-gpu detecta CUDA: `docker exec onda-v2 python -c "import onnxruntime; print(onnxruntime.get_available_providers())"`
- Si solo muestra CPUExecutionProvider → falta `onnxruntime-gpu` o CUDA libraries en el contenedor
- Fallback: usar CPUExecutionProvider (más lento pero funcional)

**Si Task 7 muestra regresión en onda-v2:**
- Verificar que onnxruntime-gpu 1.26.0 es compatible con CUDA 13.2
- Si no lo es → probar con `onnxruntime-gpu==1.27.0` o versión nightly
